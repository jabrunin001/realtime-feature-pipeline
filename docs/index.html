What this is

This pipeline answers one awkward question that breaks a lot of ML systems: does the model see the exact same features when it trains as when it serves live predictions? It generates synthetic viewing-session events (the kind a streaming service emits), runs them through Spark Structured Streaming to compute features, and writes every feature row to two places at once. Redis holds the latest values for fast serving; Delta Lake keeps the full history for training. The headline number below, a 1.0000 match rate, is the proof that those two stores agree to the decimal.

One honest caveat up front: the model itself is a placeholder. The synthetic data doesn't give it a real signal to learn (the Model tab explains why). The interesting part here is the plumbing, not the predictions.

Rebuffer events per session

How many times each session stalled to rebuffer in a five-minute window (rebuffer_count_5m), counted across the live online store. Most sessions stream clean; a long tail keeps stalling. This is exactly the kind of signal a streaming team watches to catch a bad CDN or a flaky encode.

Sessions by device

Which device each session streamed on. The generator spreads sessions fairly evenly so the features stay varied.

Sessions by network

The connection each session used. Cellular sessions tend to rebuffer more, which is the sort of thing a real model would lean on.

The features, per session

These 15 values are what the Spark job computes for every session and writes to both stores. They cover rebuffering, buffer health, bitrate, seeks, pauses, and where the session is coming from.

Model registry

The trained model lives in MLflow, versioned alongside the exact feature schema it expects. That versioning is what lets serving load the right model and refuse a mismatched one.

Online vs offline, zero skew

For a batch of sessions, the prediction served live from Redis is compared against the same prediction recomputed from the Delta history, at float tolerance. A 1.0000 match rate means the model sees identical features in training and in production. Train/serve skew is one of the quietest ways an ML system goes wrong, so proving it's zero is the whole reason this project exists.

The streaming job computes each feature row once and writes it to two places: Redis, for the low-latency lookups serving needs, and Delta Lake on MinIO, which keeps the full history so training can ask "what did this feature look like at the moment of that event." Because one computation feeds both stores, training and serving can't drift apart. That is the whole point.
